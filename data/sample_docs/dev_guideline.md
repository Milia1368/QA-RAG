# 研发规范文档 v1.8

## 一、代码提交规范

### 1.1 Commit Message 格式
遵循 **Conventional Commits** 规范：

```
<type>(<scope>): <subject>

[body]

[footer]
```

类型（type）说明：
- `feat`：新功能
- `fix`：Bug 修复
- `docs`：文档变更
- `refactor`：代码重构（非 bug 修复，非新功能）
- `test`：添加测试
- `chore`：构建过程或辅助工具变更

### 1.2 Code Review 要求
- 所有代码变更必须通过 Pull Request 合并，**禁止直接推送到 main/master 分支**
- PR 合并至少需要 **2 位** 同事 Approve
- Review 通过后，由 PR 作者自行合并
- 单个 PR 改动行数建议不超过 500 行，超出需拆分

## 二、接口规范

### 2.1 RESTful API 设计原则
1. 使用名词复数表示资源：`/users`, `/documents`
2. HTTP 方法语义：GET（查询）/ POST（创建）/ PUT（全量更新）/ PATCH（部分更新）/ DELETE（删除）
3. 统一响应格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "request_id": "uuid"
}
```

4. 错误码规范：`4xx` 客户端错误，`5xx` 服务端错误
5. 分页参数统一使用 `page` 和 `page_size`

### 2.2 接口文档
所有接口须在 Confluence 维护文档，格式包含：请求方法、路径、参数说明、响应示例、错误码列表。

## 三、测试规范

### 3.1 单元测试
- 核心业务逻辑 **单元测试覆盖率 ≥ 80%**
- 使用 pytest 框架，fixture 管理测试数据
- 禁止在单元测试中访问真实数据库或外部服务，使用 mock

### 3.2 集成测试
- 新功能上线前需完成冒烟测试
- 回归测试覆盖主干业务流程

---

## 四、部署规范

### 4.1 环境说明
| 环境 | 用途 | 发布方式 |
|------|------|---------|
| dev  | 开发联调 | 开发者自行部署 |
| staging | 测试验收 | CI/CD 自动部署 |
| prod | 生产环境 | 走上线审批流程 |

### 4.2 上线流程
1. 提交上线申请（包含变更内容、影响范围、回滚方案）
2. 技术 Leader 审批
3. 在 staging 环境验证无误
4. 灰度发布（先 5% → 25% → 100%）
5. 观察监控 30 分钟，无异常后完成上线
